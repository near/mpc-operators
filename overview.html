NEAR MPC — Network & Security Overview

How the NEAR Multi-Party Computation (MPC) signing network is structured, secured, and independently verifiable.

→ View the live operator set & threshold

NEAR's MPC network (also called Chain Signatures) lets a NEAR account produce signatures on other chains without any single party ever holding the private key. The key is split across independent operators, and a signature is only valid when a threshold of them cooperate. There is no single key and no single operator that can sign on its own — and no one, at any point, holds the full key.

Signing model

t-of-n threshold. A valid signature needs at least t of the n operators.

Operators

Independent entities. Not the NEAR Foundation / NEAR Inc. See the live list.

Key generation

Fully distributed DKG. No trusted dealer; the full key never exists anywhere.

Node integrity

Intel TDX (TEE). Attestations verified on-chain; builds reproducible.

Governance

On-chain voting by existing operators. Membership changes trigger key resharing.

Source of truth

On-chain. The v1.signer contract state is authoritative and public.

How large is the network, and what is the threshold?

The network runs a t-of-n threshold scheme: producing a signature requires cooperation from at least t of the n operators. Any group smaller than t learns nothing about the private key, and cannot forge a signature.

As of July 2026, mainnet runs 10 independent operators with a threshold of 7, with a stated target of growing to 30 operators and a threshold of 21 by the end of 2026. Because these values change as the network grows, the live operator page and the on-chain contract are always the authoritative source — this document intentionally does not hardcode them.

Who operates the nodes?

The operators are independent professional infrastructure providers — not the NEAR Foundation or NEAR Inc. The set currently includes entities such as Aurora, Everstake, Luganodes, Stakin, Black Sand, Node Monster, Staking4All, MPCH, Sovereign, and NEAR One. The live operator page shows the current members, their NEAR account IDs, and node endpoints, read directly from the contract.

How are the key shares generated?

Key shares are created through a fully distributed key generation (DKG) protocol — a variant of PedPop (“PedPop+”). There is no trusted dealer: each operator generates its own share, and the group's master public key is assembled without the full private key ever being reconstructed on any machine, at any point in time. When operators join or leave, the shares are reshared to the new set (and the threshold can be changed) without changing the public key and without exposing the key. Full protocol details: DKG documentation.

What protects a node while it is running?

Nodes are being brought under a Trusted Execution Environment requirement — Intel TDX confidential VMs — which isolate the key share from the host operating system and the operator itself. Each node publishes a remote-attestation report and runs only reproducible, on-chain-approved software builds; the contract records the approved image hashes and the attestation for each node, so anyone can verify that a node is running the expected, unmodified code. This also enforces backward secrecy: an operator that leaves the set cannot retain usable key material. TEE enforcement is being rolled out across the network, and attestation status is tracked on-chain.

How does the network change over time (governance)?

Adding or removing an operator, or changing the threshold, is an on-chain governance action: the existing operators vote on the proposed new participant set via the contract. Only once a sufficient set of current operators has voted does the network perform a key resharing to the new configuration. No single operator — and no external party — can unilaterally alter the set.

What happens if operators go offline or misbehave?

The network tolerates operators being offline: as long as at least t operators are available, signing continues. If fewer than t are available, the worst case is a failed (unproduced) signature — not a key compromise. Forging a signature or reconstructing the key would require colluding across at least t independent operators and defeating the TEE isolation on each, which is the core security assumption of the system.

Signing schemes supported

  • ECDSA on secp256k1 — Bitcoin, Ethereum & EVM chains, and others.
  • EdDSA on Ed25519 — e.g. Solana, and other Ed25519 chains.
  • Confidential key derivation on BLS12-381 — for privacy-preserving key derivation.

Verify it yourself

Don't take our word for it — the participant set and threshold are public on-chain. Query the state method of the v1.signer contract:

near contract call-function as-read-only v1.signer \
  state json-args {} network-config mainnet now

Or through the block explorer: nearblocks.io/address/v1.signer (call state under the Contract tab). The live operator page renders the same data as a table.
